Prompt 3.10: Add Production Start Support with SvelteKit Adapter Node

Task:
Fix VibeSec Lab so it can run in production mode after npm run build.

Problem:
npm run build works, but the app cannot be started as a proper production server.
The project currently appears to use SvelteKit adapter-auto.
For local production start and Cloud Run readiness, configure SvelteKit with adapter-node.

Goal:
After this fix, these commands should work:

npm run build
npm run start

The production server should serve the built SvelteKit app.

Do not rebuild the app from scratch.
Do not redesign the app.
Do not remove routes.
Do not break Gemini AI integration.
Do not expose API keys.
Do not change app features unless required for production start.

Before editing:
1. Inspect package.json.
2. Inspect svelte.config.js or svelte.config.ts.
3. Inspect vite config if needed.
4. Inspect current adapter configuration.
5. Inspect build output behavior.
6. Preserve all app routes.
7. Preserve Theme Engine.
8. Preserve Gemini server-side API routes.
9. Preserve environment variable usage.

Routes that must remain valid:
- /
- /apps
- /apps/threat-runner
- /apps/analyst-portfolio-builder

Implementation requirements:
1. Install adapter-node:
   npm install -D @sveltejs/adapter-node

2. Update SvelteKit config:
   Replace adapter-auto with adapter-node.

   Example:
   import adapter from '@sveltejs/adapter-node';

   export default {
     kit: {
       adapter: adapter()
     }
   };

3. Update package.json scripts:
   Ensure scripts include:

   "dev": "vite dev",
   "build": "vite build",
   "preview": "vite preview",
   "check": "svelte-kit sync && svelte-check --tsconfig ./tsconfig.json",
   "start": "node build"

4. Ensure production server uses:
   PORT environment variable if provided.

5. Do not hardcode port.
6. Do not expose GEMINI_API_KEY.
7. Do not move server-only Gemini helper to client code.
8. Do not convert app to static adapter because Gemini API routes need server runtime.
9. Keep all API routes functional in production.

Environment requirements:
- GEMINI_API_KEY must remain server-side.
- GEMINI_TEXT_MODEL can remain optional.
- GEMINI_CHEAP_MODEL can remain optional.
- Do not commit real .env secrets.
- If needed, add or update .env.example with placeholders only:

GEMINI_API_KEY=
GEMINI_TEXT_MODEL=gemini-2.5-flash
GEMINI_CHEAP_MODEL=gemini-2.5-flash-lite

- npm run build
- npm run start
- PORT is provided by Cloud Run
- GEMINI_API_KEY should be configured as environment variable or secret
- Do not expose GEMINI_API_KEY to client
