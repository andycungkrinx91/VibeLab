# VibeSec Lab by Andy — Product Requirements Document

## 1. Product Overview

**VibeSec Lab by Andy** adalah interactive cybersecurity playground berbasis **SvelteKit + Tailwind CSS + Magic UI-style animation**. Produk ini berisi **10 mini-apps** bertema security system, CVE intelligence, patch planning, dan incident response.

Project ini dibuat untuk demo vibe coding yang lebih matang: bukan hanya tampilan keren, tetapi punya PRD, scope jelas, route map, AI integration plan, QA plan, dan deployment path ke Google Cloud Run.

VibeSec Lab harus terasa seperti **cyber terminal command center**: modern, cinematic, command-line inspired, banyak animasi ala hacking, tetapi tetap premium, readable, responsive, dan demo-ready.

## 2. Core Narrative

### Security Flow

```txt
Detect → Triage → Patch → Verify → Deploy
```

### Build Story

```txt
Think  → Security PRD + product scope
Build  → Antigravity + Gemini CLI + SvelteKit
Launch → Google Cloud Run
```

### Video Angle

Project ini menunjukkan perbedaan antara:

* **Raw vibe coding:** cepat, langsung prompt, tapi scope bisa drift.
* **Mature vibe coding:** pakai PRD, design system, route map, AI contracts, QA, dan deploy plan.

## 3. Goals

* Membuat satu SvelteKit app berisi 10 mini-apps security-themed.
* Menggunakan design command-line / hacker command center yang premium.
* Menyediakan selectable color themes.
* Mengintegrasikan Gemini AI di semua 10 apps melalui server-side endpoint.
* Menjaga output AI dalam Bahasa Indonesia.
* Fokus AI pada defensive security: mitigasi, patching, hardening, monitoring, incident response, dan edukasi.
* Membuat app demo-ready dan Cloud Run-ready.
* Menyediakan QA flow yang jelas sebelum refinement.

## 4. Non-Goals

* Tidak membuat platform SIEM sungguhan.
* Tidak membuat vulnerability scanner sungguhan.
* Tidak mengambil real CVE feed di MVP.
* Tidak membuat exploit generator.
* Tidak membuat offensive hacking tool.
* Tidak membuat auth, RBAC, database enterprise, Kubernetes, Terraform, payment system, atau compliance platform penuh.
* Tidak membuat multi-tenant SaaS.

## 5. Target Audience

* Recruiter/client yang melihat portfolio Andy.
* Developer yang tertarik pada AI-assisted security tooling.

## 8. Security and AI Principles

Gemini AI must be used for **defensive security only**.

Allowed AI outputs:

* Risk explanation
* Mitigation plan
* Patch plan
* Hardening checklist
* Monitoring suggestions
* Incident response checklist
* Stakeholder update
* Postmortem template
* Education/explanation of concepts

Disallowed AI outputs:

* Exploit instructions
* Malware guidance
* Payload generation
* Credential theft guidance
* Bypass/evasion steps
* Offensive attack playbooks

Global AI instruction:

```txt
Jawab selalu dalam Bahasa Indonesia yang jelas, ringkas, dan praktis. Fokus pada defensive security: mitigasi, patching, hardening, monitoring, incident response, dan edukasi. Jangan memberikan instruksi eksploitasi ofensif, malware, bypass, atau penyalahgunaan sistem.
```

## 9. Runtime Model Strategy

Development/coding can use Gemini Chat, Gemini CLI, or Antigravity.

Runtime app models:

| Purpose                        | Model                   |
| ------------------------------ | ----------------------- |
| Main security text AI          | `gemini-2.5-flash`      |
| Cheap fallback/simple response | `gemini-2.5-flash-lite` |

Environment variables:

```env
GEMINI_API_KEY=
GEMINI_TEXT_MODEL=gemini-2.5-flash
GEMINI_CHEAP_MODEL=gemini-2.5-flash-lite
```

Do not use public client-side API key variables.

## 10. AI Architecture

### Server-side Only

Gemini must be called only from server-side SvelteKit routes.

Client flow:

```txt
Svelte UI → /api/ai/security → Gemini server helper → Gemini API
```

### Suggested Files

```txt
src/lib/server/gemini.ts
src/routes/api/ai/security/+server.ts
src/lib/utils/ai-client.ts
```

### API Request Shape

```json
{
  "app": "cvss-calculator",
  "action": "explain-cvss-risk",
  "input": {
    "severity": "critical",
    "internetExposed": true,
    "assetCriticality": "high",
    "exploitAvailable": true
  }
}
```

### API Response Shape

```json
{
  "ok": true,
  "app": "cvss-calculator",
  "action": "explain-cvss-risk",
  "summary": "Risiko masuk kategori Critical karena...",
  "recommendations": [
    "Aktifkan mitigasi sementara",
    "Patch maksimal 24 jam",
    "Monitor log akses"
  ],
  "warnings": []
}
```

### AI UX Requirements

Every AI action must include:

* Loading state
* Success state
* Error state
* Retry option where useful
* Indonesian response
* Defensive security output
* No raw model error shown to user

## 14. Enterprise-Ready MVP Definition

For this event/demo, “enterprise-ready” means:

* Stable route architecture
* Clear app metadata
* Reusable components
* Server-side AI only
* No API key exposure
* Safe defensive AI prompts
* Strong loading/error states
* Local persistence where useful
* Responsive design
* Build/check passes
* No obvious placeholders
* Demo-ready UX

It does **not** mean:

* Real SIEM integration
* Real CVE feed
* Production database
* Auth/RBAC
* Kubernetes/Terraform
* Compliance automation
